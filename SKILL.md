# idapt CLI

Drive the idapt AI workspace from the terminal. One binary, one grammar:
`idapt <resource> <verb> [flags]` — the same command format the in-app idapt
agent uses, so anything you learn on one surface transfers to the other.

## When to use the CLI

Reach for the CLI whenever you are already in a shell — local dev, a cloud
computer (it is pre-installed there), CI, or a script:

- **Scripting / automation** — capture ids with `-o quiet`, stream `-o jsonl`.
- **Files at the terminal** — read/write Drive directly, or `idapt drive sync` /
  `idapt drive mount` for live two-way sync of a local folder.
- **Driving idapt from another agent** — a coding agent with this CLI installed
  gets the whole platform (200+ models, agents, computers) behind one command.

Not the CLI's job: controlling a human's browser UI or a desktop GUI — those are
in-app-agent-only surfaces. For typed, in-process access from JS/TS, use the SDK
(see [Related surfaces](#related-surfaces)).

## Install + auth

```bash
curl -fsSL https://idapt.ai/cli/install | bash
idapt auth login --api-key uk_...                         # paste an existing key
idapt auth login --email you@example.com --password ...   # or email + password
```

## Command format

`idapt <resource> <verb>` plus flags. Two equivalent ways to pass arguments:

- **Flags** (humans): `idapt drive read --path notes/todo.md`. Kebab-case;
  boolean flags need no value.
- **Inline JSON** (agents / complex payloads):
  `idapt drive read --json '{"path":"notes/todo.md"}'` — camelCase keys, reliable
  for multi-line strings, embedded quotes, arrays, and nested objects. The
  envelope flags `--background` and `--timeout-seconds N` stay OUTSIDE `--json`.

This is the exact grammar the in-app dispatcher accepts, so every
`idapt instructions <res>` playbook reads the same on both surfaces.

## Discovery — lazy, asymmetric

Do not memorize the surface; fetch only what you need. Two parallel doc
surfaces share one navigation shape but differ in tier:

| Surface | Scope | Answers |
|---|---|---|
| `help` (CONTRACT) | per-verb | "what args does THIS verb take?" |
| `instructions` (PLAYBOOK) | per-resource | "should I use this, and how to wield it well?" |

```bash
idapt help                          # top-level command index
idapt help <res>                    # a resource's verbs
idapt help <res> <verb>             # per-verb CONTRACT — args, types, errors
idapt instructions <res>            # resource PLAYBOOK — when/why, anti-patterns
idapt <res> <verb> --help           # contract, flag-style
idapt <res> <verb> --instructions   # playbook, flag-style (short-circuits the call)
```

Decision flow: don't know what exists → `idapt help`. Know the verb, unsure of
the args → `--help`. Know the call, unsure it is the right move → `--instructions`.

**Destructive verbs gate on the playbook.** `drive delete`, `trigger fire`,
`computer manage`, … prompt for `--confirm` on the CLI (and, in the in-app
dispatcher, refuse the first call with the playbook inlined) until you have read
`idapt instructions <res>`. Read it first.

## Conventions

- **Output** `-o {table|json|jsonl|quiet}`. `quiet` returns ids only —
  `ID=$(idapt chat create -o quiet)` for scripting.
- **Scope** `--workspace <slug>`; defaults to `personal`.
- **Auth** `--api-key uk_...` per call, or persist once via `idapt auth login`.
- **Ids** are 26-char Crockford strings. Anywhere a resource is referenced you
  may pass an id OR a friendly slug/name (resolved against `--workspace`).
  Responses always carry the id; internal UUIDs never reach the wire.

## chat — branching, reprompt, cleanup

Every chat is a TREE of messages. Three ways to extend it:

```bash
idapt chat send <chat> "<text>"                       # append a turn at the active leaf
idapt chat send <chat> "<text>" --branch-from <msg>   # branch from an earlier message
idapt chat reprompt <chat> --message <user-msg-id>    # regenerate the reply as a sibling
```

`--effort-level fast|smart|expert` is the speed-dial: it picks the auto-router
tier and seeds the behavioural system reminder. `--model <id>` overrides for one
turn (`auto` engages the router). `--no-wait` returns a pending id immediately.

Reprompt keeps the original reply — ideal for model bake-offs:

```bash
idapt chat reprompt <chat> --message <user-id> --model anthropic/claude-opus-4.7
idapt chat reprompt <chat> --message <user-id> --model openai/gpt-5.5
idapt chat messages <chat> --last 10                  # see every sibling reply
```

Cleanup ladder — prefer reversible: `archive` (hide, reversible) →
`delete` (trash, restorable) → `permanent-delete` (irreversible, trash-only).
Full playbook: `idapt instructions chat`.

## Common errors

| Symptom | Cause | Fix |
|---|---|---|
| `401` / not authenticated | no or expired key | `idapt auth login --api-key uk_...` |
| `403` / feature not enabled | resource gated for your plan | check `idapt help`; upgrade or pick another path |
| `INSTRUCTIONS_REQUIRED` | first or destructive call to a tool group | read the inlined playbook, then retry (or pass `--confirm`) |
| `not found` for a name | wrong `--workspace` | pass the right `--workspace <slug>`, or use the resource id |
| `positional argument …` | multi-line content sent via flags | use `--json '{...}'` instead of `--flag` |
